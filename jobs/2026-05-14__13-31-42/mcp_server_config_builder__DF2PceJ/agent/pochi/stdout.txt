# LlamaCloud MCP Server Config Builder ## Background LlamaCloud can be exposed as a tool to AI agents (Claude Desktop, Cursor, Windsurf, Pochi, etc.) through the Model Context Protocol (MCP). The recommended way to do this is via the official https://github.com/run-llama/mcp-server-llamacloud package. The MCP client (e.g., Claude Desktop) reads a JSON config file that declares an `mcpServers` block; inside that block, each MCP server entry describes a `command`, an ordered `args` list, and an `env` dictionary. For LlamaCloud, the `args` are an ordered sequence of `--index <name>`/`--description <text>` pairs (one pair per LlamaCloud managed index to expose), with an optional `--topK <int>` value attached to a specific index. The `env` block MUST include `LLAMA_CLOUD_API_KEY` because the MCP server reads that variable at startup; it MAY also include `LLAMA_CLOUD_PROJECT_NAME` (defaults to `Default` if missing). A well-known friction point is that the LlamaCloud API key is scoped to a single project — pairing the wrong key with the wrong project name produces a 403 from the LlamaCloud API, so teams almost always commit the project name to source control next to the rest of the MCP config. Production teams describe the LlamaCloud indices to expose in a small YAML manifest, validate it offline, and emit the final Claude/Cursor-compatible `mcpServers` JSON. You must build that fully offline, self-contained CLI utility. To prove the LlamaCloud SDK is installed and importable, the script MUST import `LlamaCloud` from `llama_cloud.client` at module top level. It MUST NOT instantiate that client and MUST NOT make any outbound HTTP request. ## Requirements * Implement a Python CLI script `build_mcp_config.py` that imports, at module top level (verbatim): `from llama_cloud.client import LlamaCloud`.
